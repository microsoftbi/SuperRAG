## 2. 核心组件

### 2.1 `BM25EmbeddingFunction`（milvus-model 包）

SuperRAG 不再使用 `rank_bm25` 库，而是使用 Milvus 官方提供的 `BM25EmbeddingFunction`。这是一个**可训练的 BM25 模型**：

- 需要先 `fit(corpus)` 计算 IDF 字典
- 然后 `encode_documents()` / `encode_queries()` 输出**稀疏向量**（scipy.sparse.csr_array）
- 稀疏向量可直接存入 Milvus Lite 的 `SPARSE_FLOAT_VECTOR` 字段

关键参数：

```python
from milvus_model.sparse import BM25EmbeddingFunction
from milvus_model.sparse.bm25 import build_default_analyzer

# 中文分词器（基于 jieba）
analyzer = build_default_analyzer(language="zh")
bm25_ef = BM25EmbeddingFunction(analyzer=analyzer)
```

`language="zh"` 使用的分词器是 **jieba**，对中文进行语义级分词（如"退款流程"→["退款", "流程"]），而非按字符切分。

### 2.2 `BM25Retriever`（模块级单例）

`backend/app/rag/bm25_retriever.py` 定义了一个**模块级单例** `_bm25_ef`，整个应用共享同一个 BM25EmbeddingFunction 实例。

```python
_bm25_ef = None   # 模块级单例

class BM25Retriever:
    def __init__(self):
        global _bm25_ef
        if _bm25_ef is None:
            analyzer = build_default_analyzer(language="zh")
            _bm25_ef = BM25EmbeddingFunction(analyzer=analyzer)
        self._ef = _bm25_ef
```

这样做的原因：`BM25EmbeddingFunction` 的 IDF 字典在 `fit()` 后**不可增量更新**，必须全部重建。全局单例确保所有地方用的是同一份索引。

### 2.3 `VectorStoreService`（Milvus Lite 封装）

`backend/app/services/vector_store.py` 管理 Milvus Lite 集合。集合 `sprag_docs` 的 schema 包含**两个向量字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INT64 PK | 自动生成 |
| `chunk_id` | VARCHAR(64) | 原始 chunk UUID |
| `text` | VARCHAR(65535) | chunk 原文 |
| **`dense`** | **FLOAT_VECTOR(2048)** | **稠密向量**（doubao-embedding, cosine） |
| **`sparse`** | **SPARSE_FLOAT_VECTOR** | **BM25 稀疏向量**（IP 内积） |
| `document_id` | INT64 | 所属文档 ID（知识库权限过滤用） |
| `document_title` | VARCHAR(512) | 文档标题 |
| `chunk_index` | INT64 | 块序号 |

两个向量字段各自有独立的索引：

```python
index_params.add_index(field_name="dense",  index_type="FLAT",           metric_type="COSINE")
index_params.add_index(field_name="sparse", index_type="SPARSE_INVERTED_INDEX", metric_type="IP")
```

## 3. BM25 索引构建

### 3.1 索引构建时机

BM25 索引在**两个时机**构建：

| 时机 | 方式 | 触发点 |
|------|------|--------|
| **服务启动** | 同步重建 | `main.py` lifespan，从 Milvus 读取所有已有文本 |
| **文档上传后** | 异步重建 | `document_processor.py` 处理完文档后调用 `rebuild_async()` |

### 3.2 启动时构建

```python
# main.py (lifespan)
bm25 = BM25Retriever()
bm25.rebuild_index(vector_store=documents.vector_store)
if bm25.initialized:
    print(f"✅ BM25 index rebuilt ({bm25.get_dim()} terms)")
```

`rebuild_index()` → `_do_rebuild()` 的流程：

```python
def _do_rebuild(self, vector_store):
    # 1. 从 Milvus 读取所有文本
    all_texts = self._get_all_texts(vector_store)
    #    → Milvus query: SELECT text FROM sprag_docs LIMIT 100000
    
    # 2. fit() 计算 IDF 字典
    self._ef.fit(all_texts)
    #    → 内部用 jieba 分词所有文本，统计 DF（文档频率）
    #    → 计算每个词的 IDF（逆文档频率）
    
    # 3. 标记初始化完成
    self._initialized = True
    # → 之后 encode_query() 才能输出有效稀疏向量
```

`fit()` 是 BM25 模型训练的**核心步骤**：
- 接收全部文档文本列表
- 对每篇文档分词（jieba）
- 统计每个词出现在多少篇文档中（DF）
- 计算 IDF = log(1 + (N - DF + 0.5) / (DF + 0.5))
- 保存 IDF 字典

### 3.3 上传后异步重建

```python
# document_processor.py
self.vector_store.add_texts(chunk_ids, texts, metadatas)  # 先写入 Milvus

if self.bm25_retriever:
    self.bm25_retriever.rebuild_async(vector_store=self.vector_store)
```

异步重建使用**后台线程**，不阻塞上传接口：

```python
def rebuild_async(self, vector_store):
    global _rebuild_pending
    with _rebuild_lock:
        if _rebuild_pending:
            return  # 已有重建任务在排队，跳过
        _rebuild_pending = True
    
    def _bg():
        global _rebuild_pending
        try:
            self._do_rebuild(vector_store)
        finally:
            _rebuild_pending = False
    
    t = threading.Thread(target=_bg, daemon=True)
    t.start()
```

**去重机制**：`_rebuild_pending` 标志防止多个上传同时触发多次重建（重建期间再上传只重建一次）。

## 4. 检索流程

### 4.1 查询时 BM25 编码

```python
# bm25_retriever.py
def encode_query(self, query: str):
    if not self._initialized:
        # BM25 未初始化（如启动时无文档），返回空稀疏向量
        from scipy.sparse import csr_array
        return csr_array(([], ([], [])), shape=(1, 0))
    return self._ef.encode_queries([query])
```

`encode_queries()` 对查询文本做：
1. jieba 分词（如"退款流程" → ["退款", "流程"]）
2. 对每个词查 IDF 字典
3. 计算 BM25 得分向量（稀疏矩阵，每个维度对应一个词）

输出是 `csr_array` 形状 `(1, dim)`，其中 `dim` 是词汇表大小（所有文档中出现的不同词数）。

### 4.2 稀疏向量在 Milvus 中的检索

Milvus 使用 `SPARSE_INVERTED_INDEX` + `IP`（内积）度量来检索稀疏向量：

```python
# vector_store.py
req_sparse = AnnSearchRequest(
    data=[query_sparse],
    anns_field="sparse",
    param={"metric_type": "IP", "params": {}},
    limit=top_k * 2,
    expr=expr,
)
```

BM25 得分 = 查询向量与文档向量的**内积**。因为 BM25 的稀疏向量天然支持内积作为相似度度量。

## 5. 混合检索（RAG 模式）

### 5.1 HybridRetriever

```python
# hybrid_retriever.py
class HybridRetriever:
    def retrieve(self, query, k=None, doc_ids=None):
        query_sparse = self.bm25_retriever.encode_query(query)
        
        if query_sparse is not None and self.bm25_retriever.initialized:
            # BM25 已就绪 → 混合检索
            results = self.vector_store.hybrid_search(query, query_sparse, ...)
        else:
            # BM25 未就绪 → 降级为纯稠密检索
            results = self.vector_store.similarity_search(query, ...)
        
        return results
```

### 5.2 Milvus hybrid_search + RRF

```python
# vector_store.py
def hybrid_search(self, query, query_sparse, k=None, where_filter=None):
    query_emb = self.llm_service.embed([query])[0]
    
    # 两个独立的 ANN 请求
    req_dense = AnnSearchRequest(
        data=[query_emb],
        anns_field="dense",
        param={"metric_type": "COSINE"},
        limit=top_k * 2,
    )
    req_sparse = AnnSearchRequest(
        data=[query_sparse],
        anns_field="sparse",
        param={"metric_type": "IP"},
        limit=top_k * 2,
    )
    
    # Milvus 内置 RRF 融合
    results = self.client.hybrid_search(
        collection_name=COLLECTION_NAME,
        reqs=[req_dense, req_sparse],
        ranker=RRFRanker(),     # ← RRF 融合
        limit=top_k,
    )
    return self._flatten_results(results)
```

**RRF（Reciprocal Rank Fusion）算法**：

$$Score(c) = \frac{1}{k + rank_{dense}(c)} + \frac{1}{k + rank_{sparse}(c)}$$

其中 $k = 60$（Milvus RRFRanker 默认值）。每篇文档在两个检索结果列表中的排名倒数相加，按总分降序取 top_k。

### 5.3 完整检索管线（Retriever）

`Retriever.retrieve()` 编排整个流程：

```
用户问题
    │
    ├─ 1. Query Rewriting（可选）
    │     原始："它多少钱？"
    │     改写："iPhone 15 价格多少？"
    │
    ├─ 2. Hybrid Retrieval（可选）
    │     稠密检索 (cosine) ──┐
    │     BM25 稀疏检索 (IP) ──┤── RRF 融合
    │                          │
    └─ 3. Re-ranking（可选，1 次 LLM 调用批量评分）
          所有结果发给 LLM → 0-10 打分 → 排序取 top_k
```

## 6. 稀疏向量与 Milvus Lite 的协作

### 6.1 写入时

文档上传时，`VectorStoreService.add_texts()` 同时计算稠密和稀疏向量：

```python
def add_texts(self, ids, texts, metadatas):
    # 1. 稠密向量（LLM embedding）
    dense_embs = self.llm_service.embed(texts)
    
    # 2. 稀疏向量（BM25）
    sparse_embs = self._compute_sparse_batch(texts)
    
    # 3. 逐行写入 Milvus
    for i, (chunk_id, text, meta) in enumerate(zip(ids, texts, metadatas)):
        row = {
            "chunk_id": chunk_id,
            "text": text,
            "dense": dense_embs[i],
            "sparse": sparse_vec if sparse_vec is not None else {},
            #               ↑ 空字典表示"无稀疏向量"
            "document_id": meta.get("document_id", 0),
            ...
        }
        self.client.insert(COLLECTION_NAME, [row])  # 逐行插入
```

**逐行插入**的原因：BM25 稀疏向量的维度在不同批次间可能不同（词表变化），Milvus 要求同一集合中 SPARSE_FLOAT_VECTOR 的维度一致。但新文档插入时 BM25 索引还没来得及重建，因此维度可能不匹配。逐行插入并用 `{}` 占位可以绕过此限制。

### 6.2 读取时

查询时，`_compute_sparse_batch` 直接用全局单例 `_bm25_ef`：

```python
def _compute_sparse_batch(self, texts):
    from app.rag.bm25_retriever import _bm25_ef
    if _bm25_ef is None:
        return [None] * len(texts)
    embs = _bm25_ef.encode_documents(texts)
    return [embs[i:i+1] for i in range(embs.shape[0])]
```

### 6.3 BM25 未初始化时的行为

| 状态 | encode_query() 返回 | 检索行为 |
|------|-------------------|---------|
| 有文档，已 fit | 有效稀疏向量 | 混合检索（稠密+稀疏） |
| 无文档，未 fit | 空 csr_array `shape(1,0)` | 降级为纯稠密检索 |
| 文档被删后 | 标记未初始化 | 同上 |
